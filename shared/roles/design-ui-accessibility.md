# Role: UI & Accessibility Designer

Use this configuration if your primary role involves UI design with a strong focus on accessibility and inclusive design.

## Focus Areas

- UI design for web products
- Accessibility compliance (WCAG 2.2 AA+)
- Inclusive design patterns
- Assistive technology testing
- Accessible component design
- Color, contrast, and visual accessibility

## Key Responsibilities

### UI Design
- Design interfaces that are beautiful AND accessible
- Apply Scandic's visual language: warmth, simplicity, clarity
- Ensure all designs work across breakpoints (320px → 1440px)
- Design all interaction states: default, hover, focus, active, disabled, loading, error

### Accessibility Integration
- Build accessibility into the design process, not as an audit after
- Annotate designs with accessibility notes (focus order, ARIA roles, alt text)
- Create accessible color palettes: all combinations meet 4.5:1 (text) and 3:1 (UI)
- Design visible focus indicators that are both functional and on-brand
- Ensure touch targets are minimum 24x24px (44x44px recommended)

### Testing & Validation
- Test designs with screen readers (VoiceOver, NVDA, TalkBack)
- Keyboard-only navigation testing for every flow
- Zoom/magnification testing at 200% and 400%
- Reduced motion testing
- Color blindness simulation (protanopia, deuteranopia, tritanopia)

### Compliance
- **WCAG 2.2 AA**: Primary standard for all digital products
- **EN 301 549**: EU accessibility standard (legally binding)
- **European Accessibility Act (EAA)**: Mandatory compliance
- Document conformance and maintain VPAT/accessibility statements

## Accessibility Checklist for Every Design

- [ ] Color contrast meets minimum ratios
- [ ] Color is not the only indicator of meaning
- [ ] Focus states are visible and meet contrast requirements
- [ ] Touch targets are adequate size
- [ ] Text is resizable without breaking layout
- [ ] Motion can be paused or respects `prefers-reduced-motion`
- [ ] All images have appropriate alt text planned
- [ ] Form fields have visible labels (not just placeholders)
- [ ] Error states are descriptive and accessible
- [ ] Reading order makes sense when linearized
- [ ] Heading hierarchy is logical
