# Scandic Design Principles

These principles guide every design decision. When in doubt, refer back to these.

## 1. Start with User Needs

- Base decisions on evidence, not assumptions
- Talk to users regularly — research is not a phase, it's continuous
- Define the problem before designing the solution
- Measure success from the user's perspective, not ours

## 2. Accessible First

- Accessibility is not an afterthought — design for it from the start
- WCAG 2.2 AA is the minimum standard
- Test with assistive technology, not just automated tools
- Consider the full spectrum: visual, motor, cognitive, auditory, and situational disabilities
- Ask: "Can everyone use this?" before "Does this look good?"

## 3. Simple by Default

- The best interface is the one you barely notice
- Remove complexity before adding instructions
- One primary action per view
- Progressive disclosure: show what's needed when it's needed
- If it needs a tutorial, it needs redesigning

## 4. Consistent, Not Uniform

- Use the design system — don't reinvent components
- Same interaction patterns for the same tasks
- Diverge intentionally, not accidentally
- Consistency builds trust and reduces cognitive load

## 5. Mobile First, Responsive Always

- Design for the smallest screen first, then expand
- Touch-friendly interactions (44px minimum targets)
- Content priority should be clear at every breakpoint
- Test at 320px, 768px, 1024px, and 1440px

## 6. Performance is a Feature

- Fast-loading pages are accessible pages
- Optimize images, minimize JavaScript, lazy-load what's below the fold
- Target: < 3 second load time on 3G connections
- Every kilobyte is a design decision

## 7. Design with Data

- Use analytics to understand real usage patterns
- A/B test when the impact is unclear
- Let go of designs that don't serve users, regardless of effort invested
- Track accessibility metrics alongside performance metrics

## 8. Inclusive Language and Content

- Plain language over jargon
- Gender-neutral terminology
- No ableist language
- Culturally aware across Nordic markets
- Error messages should help, not blame

## How to Apply These

When making a design decision:
1. Does it serve a real user need? (Principle 1)
2. Can everyone use it? (Principle 2)
3. Is it as simple as it can be? (Principle 3)
4. Does it follow existing patterns? (Principle 4)
5. Does it work on mobile? (Principle 5)
6. Is it performant? (Principle 6)
7. Can we measure its impact? (Principle 7)
8. Is the language inclusive? (Principle 8)

If the answer to any is "no," revisit the design.
